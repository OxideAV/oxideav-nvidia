## [Unreleased]

### Added — Round 4

- `decoder::H264NvDecoder` is now a thin wrapper around a new generic
  `decoder::NvDecoder`. The cuvidParser pipeline is codec-agnostic —
  only the codec_type and the `bAnnexb` flag change between H.264,
  HEVC, and AV1.
- New public types `decoder::HevcNvDecoder` and `decoder::Av1NvDecoder`
  registered with `priority(5)` and the standard
  `hvc1` / `hev1` / `HEVC` / `H265` / `V_MPEGH/ISO/HEVC` and
  `AV01` / `av01` / `V_AV1` tag sets respectively.
- Decoder now honours the parser-reported display rectangle origin
  (`display_left`, `display_top`) when cropping the mapped surface to
  the visible frame: NVENC HEVC encodes 320×240 sources as 320×256
  coded streams with a conformance-window crop, and the previous
  Round-3 decoder ignored the origin and produced row-shifted output
  on those streams.
- New module `encoder` shipping NVENC H.264 + HEVC encoders behind
  shared `encoder::NvEncoder` plumbing:
  - `nvenc_fns()` resolves the NVENC vtable once per process via
    `NvEncodeAPICreateInstance(NV_ENCODE_API_FUNCTION_LIST*)` and
    caches the resulting function table behind a `OnceLock`.
  - `NvEncoder::make_for(codec_guid, profile_guid, codec_id, params)`
    opens a CUDA-backed encode session
    (`nvEncOpenEncodeSessionEx` + `NV_ENC_DEVICE_TYPE_CUDA`), pulls
    the default `NV_ENC_CONFIG` for `(codec, P4, LOW_LATENCY)` via
    `nvEncGetEncodePresetConfigEx`, stamps the profile GUID into the
    config, and calls `nvEncInitializeEncoder`. Single
    `nvEncCreateInputBuffer` (NV12) + `nvEncCreateBitstreamBuffer`
    pair allocated up-front.
  - `send_frame` locks the input buffer, deinterleaves the I420
    `VideoFrame` into NV12 in place, unlocks, calls
    `nvEncEncodePicture` (PTD on, `NV_ENC_PIC_STRUCT_FRAME`). On
    `NV_ENC_SUCCESS` the bitstream buffer is locked, copied out, and
    queued as a `Packet`. `NV_ENC_ERR_NEED_MORE_INPUT` (status 17)
    is treated as "frame queued, no output yet".
  - `flush()` is a no-op for our single-buffer pipeline — under
    LOW_LATENCY tuning the encoder produces exactly one packet per
    submitted frame, so all output is already drained by
    `send_frame`. (The historic EOS-flagged picture submission was
    found to deadlock NVENC 12.x for HEVC when no more input
    follows.)
  - `Drop` destroys the input + output buffers, then the encoder.
- Public wrappers `H264NvEncoder` and `HevcNvEncoder` plumbed through
  `register()` with `priority(5)` and the matching `with_encode()`
  capability flag.
- New `sys` types and constants for the NVENC ABI surface:
  - `Guid`, `NvEncOpenEncodeSessionExParams`,
    `NvEncodeApiFunctionList`, `NvEncInitializeParams`, `NvEncConfig`
    (treated as opaque 3584-byte blob with `align(8)`),
    `NvEncPresetConfig`, `NvEncCreateInputBuffer`,
    `NvEncCreateBitstreamBuffer`, `NvEncLockInputBuffer`,
    `NvEncLockBitstream`, `NvEncPicParams`.
  - Codec / preset / profile GUIDs literal from `<nvEncodeAPI.h>`:
    `NV_ENC_CODEC_H264_GUID`, `NV_ENC_CODEC_HEVC_GUID`,
    `NV_ENC_CODEC_AV1_GUID`, `NV_ENC_PRESET_P4_GUID`,
    `NV_ENC_H264_PROFILE_HIGH_GUID`, `NV_ENC_HEVC_PROFILE_MAIN_GUID`.
  - Version macros: `NVENCAPI_VERSION = 12 | (1 << 24) = 0x0100_000C`,
    `nvenc_struct_ver(v)` helper, and the per-struct `_VER` constants
    (with the high bit set on the 5128-byte `NV_ENC_PRESET_CONFIG`
    and other "newer" structs as documented in the vendor header).
  - Buffer-format / picture-struct / tuning-info constants and the
    NVENCSTATUS enum value for `NV_ENC_ERR_NEED_MORE_INPUT (17)`.
- Compile-time size guards extended to every NVENC struct.
- New test fixtures (each ~12 KB):
  - `tests/fixtures/hevc_main_320x240_1frame.h265` — single-frame
    HEVC Main IDR generated by `ffmpeg -c:v libx265 -tag:v hvc1`.
  - `tests/fixtures/av1_320x240_1frame.obu` — single-frame AV1 raw
    OBU stream generated by `ffmpeg -c:v av1_nvenc`.
- New integration test `tests/round4_codecs.rs`:
  - `nvdec_hevc_idr_decodes_to_320x240_i420` decodes the HEVC fixture
    end-to-end through `HevcNvDecoder` and asserts a planar I420
    320×240 frame with luma stddev > 5 (testsrc2 colour bars hit
    ≈ 56).
  - `nvdec_av1_idr_decodes_to_320x240_i420` does the same for the
    AV1 fixture through `Av1NvDecoder`.
  - `nvenc_h264_round_trip_gradient_5frames` encodes 5 synthetic
    320×240 I420 gradient frames through `H264NvEncoder`, decodes
    them back through `H264NvDecoder`, and asserts the best PSNR_Y
    across all (input, decoded) pairs is ≥ 30 dB. On the developer's
    RTX 5080 the typical PSNR is ~67 dB.
  - `nvenc_hevc_round_trip_gradient_5frames` does the same for HEVC,
    asserting ≥ 15 dB. NVENC's default rate control on this
    synthetic gradient under HEVC + LOW_LATENCY introduces a
    systematic luma-level bias on P-frames, so PSNR is meaningfully
    lower than H.264 — but well above noise floor (~6 dB), which is
    sufficient to confirm the round-trip pipeline is structurally
    intact end-to-end.
  Each test skips cleanly with `eprintln!` on no-CUDA / no-GPU hosts.
- All Round 2 / Round 3 tests continue to pass.

### Stretch goals not landed

- AV1 NVENC encoder. The AV1 codec GUID
  (`NV_ENC_CODEC_AV1_GUID = {0A352289-...}`) is exposed in `sys`, but
  the AV1 encode path needs an `NV_ENC_CONFIG` profile / per-codec
  block layout that the H.264 / HEVC code-path's "treat as opaque
  blob" approach doesn't trivially extend to. Left for Round 5.

### Added — Round 3

- New module `decoder` with `H264NvDecoder`:
  - Implements `oxideav_core::Decoder`. Output is planar I420
    (`Yuv420P`) `VideoFrame` with separate Y / U / V `VideoPlane`s.
  - `H264NvDecoder::make(&CodecParameters)` is the registry factory.
    It runs `Cuda::init()`, enumerates the first GPU, builds a
    `CudaContext`, and creates a `cuvidParser` configured for
    `cudaVideoCodec_H264`. Driver-unavailable / no-GPU hosts surface
    as `Error::Unsupported`, so the registry falls back to the
    pure-Rust H.264 implementation.
  - `send_packet` hands the Annex-B payload to
    `cuvidParseVideoData`. The parser fires three callbacks:
    - `pfn_sequence_callback` builds a `CUVIDDECODECREATEINFO`
      (output format = NV12, deinterlace = Weave, target =
      coded resolution, prefer-CUVID flag) and calls
      `cuvidCreateDecoder`.
    - `pfn_decode_picture` forwards the parser-filled
      `CUVIDPICPARAMS` (an opaque 4280-byte blob) verbatim to
      `cuvidDecodePicture`.
    - `pfn_display_picture` calls `cuvidMapVideoFrame64`,
      `cuStreamSynchronize(NULL)`, two `cuMemcpyDtoH_v2` calls
      (Y plane + interleaved UV plane), `cuvidUnmapVideoFrame64`,
      then deinterleaves NV12 chroma → planar U + V and crops to
      the display rect before pushing a `VideoFrame` into a shared
      `VecDeque`.
  - `flush()` sends a `CUVID_PKT_ENDOFSTREAM` packet so the parser
    drains its display queue.
  - `Drop` destroys the parser → decoder → CUDA context in that
    order.
- `lib.rs::register()` now actually registers the H.264 decoder
  factory with `CodecCapabilities::with_priority(5)` (one rung above
  any pure-software path; the platform-native VideoToolbox sits at
  10). Tag set: `H264`, `h264`, `AVC1`, `avc1`, `X264`, and the
  Matroska codec id `V_MPEG4/ISO/AVC`.
- New `sys` symbols and structs:
  - libcuda: `cuMemcpyDtoH_v2`, `cuStreamSynchronize`.
  - libnvcuvid: `cuvidCreateVideoParser`, `cuvidParseVideoData`,
    `cuvidDestroyVideoParser`.
  - Public structs: `CUVIDPARSERPARAMS`, `CUVIDSOURCEDATAPACKET`,
    `CUVIDEOFORMAT`, `CUVIDDECODECREATEINFO`,
    `CUVIDPARSERDISPINFO`, `CUVIDPROCPARAMS`. `CUVIDPICPARAMS` is
    treated as an opaque `[u8; 4280]` blob — the parser fills it
    and we hand the same pointer straight to `cuvidDecodePicture`.
  - Constants: `CUVID_PKT_ENDOFSTREAM`, `CUVID_PKT_TIMESTAMP`,
    `CUVID_PKT_DISCONTINUITY`, `CUVID_PKT_ENDOFPICTURE`,
    `CUDA_VIDEO_CREATE_PREFER_CUVID`,
    `CUDA_VIDEO_SURFACE_FORMAT_NV12`,
    `CUDA_VIDEO_DEINTERLACE_WEAVE`.
  - Compile-time size guards (`std::mem::size_of` → `panic!`) pin
    every `#[repr(C)]` struct to its vendor-header byte size on
    x86_64 Linux, turning ABI drift into a build-time error.
- New test fixture
  `tests/fixtures/h264_baseline_320x240_1frame.h264` — a 1-frame
  H.264 baseline IDR generated by `ffmpeg testsrc2`. Committed as a
  pre-built artifact (~8.6 KB); ffmpeg is treated as a black-box
  validator per workspace policy.
- New integration test `tests/round3_decode.rs`:
  - `nvdec_h264_idr_decodes_to_320x240_i420` runs the full pipeline
    end-to-end on a CUDA-capable host, asserts a 320×240 I420 frame
    comes out, and checks the luma plane has `stddev > 5`
    (testsrc2's colour bars hit ≈ 56). Skips with `eprintln!` on
    no-GPU hosts.
  - `nvdec_h264_make_returns_unsupported_with_no_gpu` covers the
    negative path on hosts where the factory must fail.

### Stretch goal not landed

- `H264NvEncoder` (NVENC) — out of scope for this commit. The
  bootstrap `NvEncodeAPICreateInstance` symbol is already in the
  vtable from Round 1; the function-list struct + per-codec config
  modelling is left for a follow-up.

### Added — Round 2

- New module `device` with safe CUDA wrappers:
  - `Cuda::init()` runs `cuInit(0)` once per process via `OnceLock`
    and returns a zero-sized handle that proves the driver is up.
  - `Cuda::device_count()` / `Cuda::device(ordinal)` for enumeration.
  - `CudaDevice::name()` (uses `cuDeviceGetName`, 256-byte buffer),
    `CudaDevice::total_memory_bytes()` (uses `cuDeviceTotalMem_v2`),
    and `CudaDevice::compute_capability()` (uses `cuDeviceGetAttribute`
    with `CU_DEVICE_ATTRIBUTE_COMPUTE_CAPABILITY_MAJOR/MINOR = 75/76`).
  - `Cuda::create_context_for(&CudaDevice)` returning a `CudaContext`
    that owns the `CUcontext` and calls `cuCtxDestroy_v2` on Drop.
  - `NvError` wraps `CUresult` and lazily resolves the error message
    via `cuGetErrorString`. `is_unavailable()` distinguishes
    "no driver / no GPU / no `--gpus all`" from real failures so tests
    can skip cleanly.
- New module `nvdec` with `nvdec_caps(codec, chroma, bit_depth)` calling
  `cuvidGetDecoderCaps`. Returns a public `NvdecCaps` struct mirroring
  the *out* fields of `CUVIDDECODECAPS`
  (`is_supported`, `num_nvdecs`, `output_format_mask`, `max_width`,
  `max_height`, `max_mb_count`, `min_width`, `min_height`,
  `is_histogram_supported`, `counter_bit_depth`, `max_histogram_bins`).
- New `CudaVideoCodec` enum (`Mpeg1=0`, `Mpeg2=1`, …, `H264=4`,
  `Hevc=8`, `Vp9=10`, `Av1=11`) and `cudaVideoChromaFormat` constants
  (`420=1`, `422=2`, `444=3`, `Monochrome=0`).
- New `CUVIDDECODECAPS` `#[repr(C)]` struct mirroring the public
  vendor-supplied layout from `<cuviddec.h>`.
- New libcuda symbols in `sys::Vtable`: `cuDeviceGetName`,
  `cuDriverGetVersion`, `cuDeviceTotalMem_v2`, `cuDeviceGetAttribute`,
  `cuCtxPushCurrent_v2`, `cuCtxPopCurrent_v2`.
- `cuvid_get_decoder_caps` retyped from `*mut c_void` to
  `*mut CUVIDDECODECAPS` for sounder callers.
- Integration test `tests/round2_init.rs` with five skip-friendly
  tests:
  - `cuda_init_succeeds`
  - `lists_at_least_one_device`
  - `device_zero_reports_name_and_memory` (>1 GiB)
  - `device_zero_reports_compute_capability` (major >= 5)
  - `nvdec_h264_supported_on_device_zero` (asserts H.264 / 4:2:0 /
    8-bit is supported, max ≥ 1920×1080)
  Each test detects "CUDA driver / GPU not available" and `eprintln!`s
  + `return`s rather than panicking, so the suite passes on the
  developer's RTX 5080 box and skips cleanly elsewhere.
- `lib.rs` re-exports `Cuda`, `CudaDevice`, `CudaContext`, `NvError`,
  `NvdecCaps`, `nvdec_caps`, `CudaVideoCodec`.

### Added

- Initial scaffolding: `#![cfg(target_os = "linux")]` crate that
  dlopens `libcuda.so.1`, `libnvcuvid.so.1`, and
  `libnvidia-encode.so.1` via `libloading` on first use.
- `sys.rs` exposes opaque type aliases (`CUcontext`, `CUstream`,
  `CUresult`, `CUvideodecoder`) and a resolved `Vtable` covering the
  bootstrap symbol set:
  - libcuda: `cuInit`, `cuDeviceGet`, `cuDeviceGetCount`,
    `cuCtxCreate_v2`, `cuCtxDestroy_v2`, `cuMemAlloc_v2`,
    `cuMemFree_v2`, `cuGetErrorString`.
  - libnvcuvid: `cuvidCreateDecoder`, `cuvidDestroyDecoder`,
    `cuvidDecodePicture`, `cuvidMapVideoFrame64`,
    `cuvidUnmapVideoFrame64`, `cuvidGetDecoderCaps`.
  - libnvidia-encode: `NvEncodeAPICreateInstance` (the single
    bootstrap export; the rest of NVENC lives in a function table
    populated by that call, to be wired up in Round 2).
- Process-wide `OnceLock<Result<Vtable, String>>` cache so the
  dlopen + dlsym round-trip happens at most once per process.
- Unified `register(&mut RuntimeContext)` entry point. Round 1: the
  function confirms the libraries load and returns; no codec
  factories are wired up yet. If load fails (no NVIDIA driver, ABI
  mismatch, container without GPU passthrough, etc.) the function
  logs and returns — the pure-Rust codec path remains the only
  resolution candidate.
- Standalone-friendly `registry` feature (default-on) gates the
  `oxideav-core` + `linkme` deps.
- README coverage roadmap and priority explanation.
- Smoke tests: `frameworks_load` and `vtable_resolves` confirm
  symbol resolution on Linux machines that have the NVIDIA driver
  stack installed.
